# Project B v0.4.3 / CCF 2.1: Quality, multiplicity confidence, and external clonality evidence

This release adds the P0-P1 CCF optimization layer on top of Project B v0.4.2 P1.
It is designed for computational triage and candidate prioritization, not clinical
clonality diagnosis.

## What changed

### 1. CCF input QC

`ccf-2` now writes `ccf_input_qc.tsv` next to the main CCF output by default.
It summarizes:

- purity and purity confidence
- ploidy
- CNV source and CNV quality
- segment count
- CNA/LOH genome fraction when segment coordinates are available
- major/minor copy-number availability
- whether CCF estimates are ready, approximate, or limited

### 2. Multiplicity confidence

The main CCF table now includes:

- `multiplicity_candidates`
- `multiplicity_best`
- `multiplicity_confidence`
- `multiplicity_ambiguity`

This prevents overinterpreting a single `multiplicity_best` when several copy-number
multiplicities are biologically plausible.

### 3. Clonality confidence

The main CCF table now includes:

- `clonality_confidence`
- `vaf_ci_low` / `vaf_ci_high`
- `probability_ccf_gt_0_8`
- `probability_ccf_lt_0_3`

The probability fields are coarse interval-derived summaries, not Bayesian
posterior probabilities. They are meant to make uncertainty explicit.

### 4. Event-type-aware CCF methods

`ccf_method` now distinguishes:

- `SNV_INDEL_COPY_NUMBER_AWARE`
- `SV_BREAKPOINT_APPROX`
- `WES_SV_CAPTURE_LIMITED_APPROX`
- `JUNCTION_APPROX`
- `RNA_ONLY_UNRESOLVED`
- `COPY_NUMBER_AWARE_APPROX`

RNA-only fusion/splice events are not assigned a pseudo-CCF.
WES SV candidates remain low-confidence by default.

### 5. External clonality / SVclone adapters

`ccf-2` accepts optional external evidence:

```bash
neoag ccf-2 \
  --events parsed/raw_events.tsv \
  --purity purity.tsv \
  --cnv cnv_segments.tsv \
  --external-clonality pyclone_vi.tsv \
  --svclone svclone.tsv \
  --out clonality/ccf_lite.tsv
```

Supported external tables are intentionally permissive. For external clonality,
accepted columns include:

- `event_id`, `mutation_id`, `variant_id`
- `cellular_prevalence`, `ccf`, `ccf_best`
- `cellular_prevalence_low`, `ccf_low`, `ccf_ci_low`
- `cellular_prevalence_high`, `ccf_high`, `ccf_ci_high`
- `cluster_id`, `cluster_assignment_probability`
- `tool`

For SVclone-like tables, accepted columns include:

- `event_id`, `sv_event_id`, `sv_id`
- `sv_ccf`, `ccf`, `cellular_prevalence`
- `sv_ccf_low`, `sv_ccf_high`
- `sv_copy_number`
- `confidence`

### 6. Conflict and cluster sidecars

`ccf-2` now writes:

- `ccf_conflicts.tsv`
- `ccf_cluster.tsv`

Conflicts are reported when internal and external CCF estimates diverge materially
or produce discordant clonality labels. SVclone evidence is preferred for SV events
but remains explicitly labeled as external evidence.

## Important boundaries

- CCF is inferred from bulk sequencing; it is not a single-cell truth.
- SNV/InDel CCF is more interpretable than SV breakpoint approximation.
- WES SV CCF remains capture-limited and low-confidence by default.
- RNA-only fusion/splice events should use RNA abundance support, not DNA CCF.
- External deconvolution tools should be used as optional evidence, not hidden overrides.

## Outputs

Main output:

- `ccf_lite.tsv` with legacy and CCF 2.1 columns

Sidecars:

- `ccf_input_qc.tsv`
- `ccf_conflicts.tsv`
- `ccf_cluster.tsv`

## Example

```bash
PYTHONPATH=src python -m neoag.cli ccf-2 \
  --events data/events.tsv \
  --purity data/purity.tsv \
  --cnv data/cnv_segments.tsv \
  --external-clonality data/pyclone_vi.tsv \
  --svclone data/svclone.tsv \
  --out results/clonality/ccf_lite.tsv
```
